Ocelot UI Roadmap

A full assessment of what the library needs to become production-ready, compared against the top Vue and React UI frameworks, grounded in UX law, and mapped to a phased delivery plan.

April 2026 v0.0.21 23 components today 53 registered icons

1 Current State Assessment

What exists, what's strong, and what's missing.

What exists — 23 components

CategoryComponents

FormsInput (only one)

Genuine strengths

Accessibility-first

Axe runs on every Storybook story. Live regions on Toast, focus management via useFocusMemory, semantic HTML throughout. This is ahead of most open-source libraries.

Tailwind v4 peer dep

Consumers own their CSS. Zero style leakage. OUI classes tree-shake naturally. The @source pattern in tailwind.css is correct and forward-looking.

Advanced media

LQIP blur-up on Image, multi-format video (HTML5/YouTube/Vimeo), PhotoSwipe lightbox, and QR code generation are well above the average component library.

FloatingPanel

Full Floating UI integration — flip, shift, arrow, all 12 placements. Click/hover/all interaction modes. Most libraries charge for a panel this capable.

Composable architecture

useToast, useLightboxRegistry, useFocusMemory are independently usable. Good separation of concerns.

Testing infrastructure

Vitest + Playwright + Axe. The test-runner hook that injects Axe before every story is a particularly strong pattern worth preserving.

Critical gaps

  • No navigation patterns — Tabs, Breadcrumb, Pagination, Steps are all missing.
  • Only one form component. No Select, Checkbox, Radio, Switch, or Textarea.
  • No inline feedback (Alert/Banner). Toast is not a substitute.
  • No data display — Table, Avatar, Timeline, Stats card, Progress.
  • Icon system is hard-coded to 53 icons. No consumer extensibility. No component-as-prop support.
  • Button has no icon support — treated as first-class by every major framework.
  • Card uses a flat prop API where composable slots are expected.
  • No documented theming or design token story.

2 Competitive Landscape

What the best libraries do — and what OUI should adopt from each.
TailwindUI
Design bar

The gold standard for visual design and component composition. Every pattern ships with dark mode and responsive breakpoints. Named slots are used consistently across all patterns — #header, #footer, #actions.

Key lesson: Components must look finished out of the box. Named slot conventions should be consistent across the entire library.

Headless UI (Tailwind Labs)
A11y reference

Fully headless, behaviour-only. The definitive reference for keyboard navigation and ARIA correctness — especially on Listbox, Combobox, Menu, and Dialog.

Key lesson: Complex interactive components need a complete keyboard spec before implementation begins, not after.

Radix UI / shadcn
Icon pattern

Icons are never bundled — consumers pass components as props. Sub-component APIs (Dialog.Root, Dialog.Trigger, etc.) make ARIA relationships automatic and reduce the surface area of the root component.

Key lesson: Icon-as-prop is correct for a library. Sub-component APIs are cleaner than monolithic components with many props.

Mantine
Slots model

Uses leftSection/rightSection slot names consistently across all input components. Form integration via useForm composable propagates error states via context. CSS custom property theming at the colour level.

Key lesson: Consistent compositional slot naming creates a predictable API. provide/inject for form context is the correct Vue pattern.

Nuxt UI v3 / PrimeVue
Vue-native

Nuxt UI v3 is Tailwind v4 based — the closest direct competitor. PrimeVue ships CSS custom property theming with a UI builder. Both show that consumers expect some customisation path, even if limited.

Key lesson: Document at minimum a `:class` override pattern and a small set of CSS custom properties (radius, shadow, transition).

✦
Patterns OUI should adopt: Icon-as-prop. Consistent slot naming (#prefix, #suffix, #header, #footer, #actions, #empty, #icon). Sub-component APIs for complex components. Dark mode required everywhere. Full keyboard specs before implementation.

3 Icon Library — The Critical Problem

The current registry of 53 hand-registered icons does not scale. Here is the full solution.
⚠
The problem: Adding icons requires editing library source + rebuilding + publishing. Consumers cannot add their own icons. Importing all 5,000+ Tabler icons would add ~2MB to the bundle. The current model cannot continue.

Solution: Hybrid icon prop

Change the icon prop on Icon.vue to accept either a string (resolved against the registry) or a Vue component (rendered directly). This is backwards-compatible — all existing string usage continues to work.

// String — current API, backwards-compatible
<OuiIcon icon="Check" size="lg" />

// Component — new, consumer controls their own bundle
import { IconBrandGithub } from '@tabler/icons-vue'
<OuiIcon :icon="IconBrandGithub" size="lg" />

The Icon.vue implementation uses typeof icon === 'string' to decide between registry lookup and <component :is="icon" />.

Consumer extensibility — registerIcons()

Export a function that augments the registry at app initialisation. Consumers use string names throughout their app without importing the component at every usage site.

// consumer's main.js
import { registerIcons } from 'ocelot-ui'
import { IconBrandGithub, IconWorld } from '@tabler/icons-vue'

registerIcons({ BrandGithub: IconBrandGithub, World: IconWorld })

TypeScript

  • Export IconName as a union of all built-in registry keys.
  • Export IconProp = IconName | Component — the type for any prop that accepts an icon.
  • Document module augmentation for consumers to extend IconName with their registered icons and get autocomplete.

Expand the built-in registry: 53 → ~120

4 Theming & Design Tokens

Consumers currently have no documented way to customise OUI's visual appearance.

OUI uses Tailwind utility classes directly. A consumer who wants a different border radius or transition speed must guess which classes to override — and an OUI upgrade may silently break them. A small, documented set of CSS custom properties fixes this.

Recommended: CSS custom properties for structural values

/* additions to tailwind.css (the file consumers already import) */
:root {
  --oui-radius:          0.375rem;
  --oui-radius-sm:       0.25rem;
  --oui-radius-lg:       0.5rem;
  --oui-radius-full:     9999px;
  --oui-shadow:          0 1px 3px 0 rgb(0 0 0 / 0.1);
  --oui-shadow-md:       0 4px 6px -1px rgb(0 0 0 / 0.1);
  --oui-ring-width:      2px;
  --oui-ring-offset:     2px;
  --oui-transition:      150ms;
  --oui-font-sans:       inherit;
}
ℹ
What NOT to expose: Do not create --oui-primary-* colour tokens. Consumers configure Tailwind colours themselves — OUI should use their palette, not override it. Focus the token layer on structural values (radius, shadow, timing) not colour.

Storybook: Design System section

Add a new top-level Storybook section with MDX pages covering: Colour Palette (expand existing), Typography scale, Spacing & Sizing, Border Radius & Shadows (with override examples), and Motion & Transitions.

5 Existing Component Improvements

What to add or fix in the 23 components that already exist.
Button
Update+ ButtonGroup

No icon support is the biggest gap — every major framework treats icon+label as first-class. The small variant may violate the 44×44px minimum touch target (Fitts's Law).

  • Add prefixIcon and suffixIcon props — accept IconProp (string or Component).
  • Add iconOnly prop — square aspect ratio, enforced aria-label.
  • Add ButtonGroup component — removes interior border radii, collapses margins, cascades size and variant to children.
  • Audit small variant touch target — must be ≥ 44×44px (invisible padding if needed).

Story Icon-only buttons, icon+label (prefix+suffix), ButtonGroup (horizontal+vertical), full size × variant matrix.

Badge
Update
  • Add dot variant — coloured circle, no label. Sizes: sm/md/lg (6/8/10px).
  • Add removable prop — appends a close <button>, emits @remove. Button must have aria-label="Remove [label]" (WCAG 2.5.3).
  • Add outline variant — border only, transparent background.
  • Add truncate prop — for dynamically generated labels.

Story Dot (all colours, all sizes), removable (play function tests @remove event), outline, truncated label, badge in context inside Card and Avatar.

Card
Refactor (backwards-compatible)

The flat-prop API becomes limiting quickly. TailwindUI, Mantine, and Nuxt UI all use slot-based card sub-components.

  • Add named slots: #media, #header, #content, #footer, #actions. Keep existing props as convenience wrappers that populate these slots.
  • Add clickable prop — makes entire card interactive with hover/focus states.
  • Add selected prop — visual selected state (ring + tick) for selection UIs.
  • Add variant prop: default, bordered (no shadow), flat (no shadow, no border).

Story All slot combinations, clickable, selectable (play function toggles), card grid layouts (3-up, 4-up).

Accordion
Update+ AccordionGroup
  • Add AccordionGroup — enforces exclusive-open (only one item open at a time) via provide/inject.
  • Add variant prop: default (bordered), flush (dividers only), contained (boxed background).
  • Add #expandIcon slot to override the default chevron.
  • Consider switching from native <details> to manual aria-expanded/aria-controls for consistent screen reader behaviour.

Story AccordionGroup exclusive mode, all variants, with custom expand icon, with rich content.

Dialog
Update+ ConfirmDialog
  • Add #title slot with automatic aria-labelledby binding (visible title is preferred over aria-label — WCAG 2.4.6).
  • Add size prop: sm (400px), md (560px), lg (768px), xl (1024px), fullscreen.
  • Add #description slot linked via aria-describedby.
  • Add ConfirmDialog sub-component with Promise-based API: await confirmRef.value.open({ title, description, confirmLabel, confirmColor }).

Story With title slot, all sizes, with scrolling content, with form (focus trap test), ConfirmDialog (play function confirms and cancels).

FloatingPanel → Tooltip + Popover
Named exports

FloatingPanel is capable but consumers expect conventional names. The ARIA semantics differ enough to warrant distinct exports.

  • OuiTooltip — preset: hover, role="tooltip", 300ms delay, max-width. Props: content, placement, delay, disabled.
  • OuiPopover — preset: click, role="dialog", no focus trap. Has close button in header. Slot-based content.
  • Add delay prop to base FloatingPanel for hover interactions.

Story Separate Tooltip story (on icon button, on disabled wrapper, multiline). Separate Popover story (with form content, with actions).

Toast
Update
  • Add action option to useToast().add(): { label: string, onClick: () => void }. Renders an inline text button ("Undo", "View", "Retry").
  • Add icon override option — pass custom IconProp per toast.
  • Add onClose callback option.
  • Export useToast from the library root — it currently requires importing from an internal path.

Story Toast with action (play function clicks action), all types with custom icon, persistent toast, rapid stacking test.

Sidebar
+ Nav sub-components
  • Add SidebarNav — <nav aria-label="..."> wrapper with consistent spacing.
  • Add SidebarNavGroup — labelled group of nav items with optional collapsible header.
  • Add SidebarNavItem — href, active (applies aria-current="page"), icon (IconProp), badge count, disabled.
  • Add width prop to Sidebar.

Story Full nav structure with groups, active state, badge count, mobile pattern (scrim + slide), programmatic open/close.

6 Missing Components

Grouped by priority tier. Tier 1 is essential for a usable library.

Tier 1 — Essential

Alert / Banner
Tier 1New

Inline, embedded feedback — distinct from Toast. Alert is persistent and in-layout (form errors, system notices). Toast is ephemeral and floating.

Props: type (info/success/warning/error), title, dismissible, variant (solid/subtle/outline)
Slots: #icon, #title, default (description), #actions
ARIA: role="alert" for error/warning (assertive); role="status" for info/success (polite); no role for static alerts.

Story All 4 types × 3 variants, dismissible (play function), with action button, inside form layout.

Avatar / AvatarGroup
Tier 1New

Used in nearly every product — profile headers, comment threads, team lists, user tables.

Fallback priority: image → initials → generic User icon
Props: src, alt, initials (1–2 chars), size (xs–2xl), color (for initials bg), shape (circle/square)
AvatarGroup: max (visible count), size — shows +N overflow badge, overlapping layout.

Story With image, initials (all colours), icon fallback, all sizes, square, AvatarGroup with overflow, AvatarGroup with tooltip names.

Tabs / TabPanels
Tier 1New

One of the most fundamental navigation primitives. Must follow ARIA APG exactly — deviating from the expected keyboard model will cause complaints immediately.

Sub-components: Tabs (root), TabList, Tab, TabPanel
Props: v-model (active tab ID), orientation (horizontal/vertical), variant (underline/pill/bordered/lifted), activation (auto/manual), lazy

KeyAction
→ / ↓Move focus to next tab
← / ↑Move focus to previous tab
HomeMove focus to first tab
EndMove focus to last tab
Enter / SpaceActivate focused tab (manual mode)

Story All 4 variants, vertical, tab with icon+badge, disabled tab, controlled (v-model), lazy mounting, keyboard navigation play function.

Progress (Bar + Circle)
Tier 1New

ProgressBar props: value (0–max), max (default 100), size (sm/md/lg), color, animated (indeterminate state), label, showValue
ProgressCircle props: same value/max/color/size + strokeWidth, showValue (centred in ring)
ARIA: role="progressbar", aria-valuenow, aria-valuemin, aria-valuemax. Omit aria-valuenow for indeterminate.

Story Determinate (all colours, sizes), indeterminate (animated), with label, ProgressCircle.

Breadcrumb
Tier 1New

Props: items[] ({label, href?}), separator (string or Component), maxItems (collapse middle items)
Semantic: <nav aria-label="Breadcrumb"><ol> with aria-current="page" on last item.

Story Basic 3-level, custom icon separator, collapsed long path (5+ levels), expand collapsed play function.

Pagination
Tier 1New

Props: v-model (current page), total, siblings, boundaries, size, compact (prev/next only)
ARIA: <nav aria-label="Pagination">, aria-current="page", descriptive aria-label on each button.
Miller's Law: Show maximum 7 page numbers; collapse the rest with an ellipsis.

Story Basic, compact, large dataset (100+ pages), size variants, click navigation play function.

Steps / Stepper
Tier 1New

For multi-step forms, wizards, and checkout flows.
Sub-components: Stepper (root), Step
Step props: label, description, status (complete/active/pending/error), icon
ARIA: role="list" on Stepper, aria-current="step" on active step.

Story Horizontal (all statuses), vertical, linear navigation play function, error state, with description.

Tier 2 — Important

Dropdown Menu
Tier 2New

Distinct from FloatingPanel — a fully ARIA-compliant menu pattern with exhaustive keyboard navigation. This is one of the most complex components to implement correctly.

Sub-components: DropdownMenu, DropdownMenuTrigger, DropdownMenuContent, DropdownMenuItem, DropdownMenuCheckboxItem, DropdownMenuSeparator, DropdownMenuLabel, DropdownMenuSub

MenuItem props: label, icon (IconProp), shortcut (display string), disabled, destructive (red), href
Keyboard: Arrow Up/Down navigate items, Home/End jump to boundaries, A–Z character search, Arrow Right/Left for sub-menus, Escape closes and returns focus to trigger.

Story Basic, with icons+shortcuts, with checkbox items, with destructive item, with nested sub-menu, keyboard navigation play function.

Table
Tier 2New

Semantic sub-components — not a DataTable with built-in filtering. That complexity stays with the consumer; OUI provides the accessible structure.

Sub-components: Table, TableHead, TableBody, TableFoot, TableRow, TableHeader (th), TableCell (td)
Table props: striped, hoverable, bordered, size, stickyHeader, caption
TableHeader: sortable, sortDirection, emits sort. ARIA: aria-sort.
Slots: #empty (integrate with EmptyState), #loading (integrate with Placeholder skeleton).

Story Basic, striped, sortable columns (play function), empty state slot, loading skeleton slot, sticky header, responsive horizontal scroll.

Stats / Metric Card
Tier 2New

Essential for dashboards. The value must be visually dominant; delta is subordinate (Aesthetic-Usability Effect).

Props: label, value, delta (positive=green, negative=red), deltaLabel, icon (IconProp), variant (card/bordered/inline), loading

Story Positive/negative/neutral delta, with icon, loading state, 4-up grid layout, all variants.

Empty State
Tier 2New

A polished empty state prevents pages from feeling broken (Aesthetic-Usability Effect).
Props: title, description, variant (page/inline/compact)
Slots: #icon, #title, #description, #actions

Story Page empty (with icon + action), table empty (inline), search result empty, compact.

Timeline · Code Block
Tier 2New

Timeline: Timeline + TimelineItem. Props: label, date, description, icon, status. Semantic: <ol> list. Connector line is aria-hidden.

Code Block: Provides the chrome (copy button, filename header, line numbers, scroll container). Does not bundle a syntax highlighter — consumers pass pre-highlighted HTML via #highlighted slot (Shiki/Prism compatible).

CommandPalette (⌘K) Chip / Tag (interactive badges) Callout (editorial) SkipLink (WCAG 2.4.1)

Form context architecture (provide/inject)

A Form wrapper propagates validation state to all child form components. FormField is the key composition unit:

Form  (provide: validationState, registerField, submit)
  └── FormField  (inject validationState, provides ARIA links)
        ├── Label          (for="[input-id]")
        ├── Input          (id, aria-describedby)
        ├── HelperText     (always visible)
        └── ErrorMessage   (only visible in invalid state)

Validation states

All form components must support four states consistently: idle → valid (green) → invalid (red) → warning (amber). States propagate from Form via provide or can be set directly via prop.

✦
Tesler's Law in action: Form complexity cannot be reduced, only shifted. FormField absorbs the ARIA linking (for/id, aria-describedby pointing to helper + error) so consumers write clean, simple template code.

8 Cross-Cutting Technical Concerns

Things that affect every component, not just one.

SSR / Nuxt compatibility

Many consumers will use OUI with Nuxt. Current risks:

  • Browser API access — FloatingPanel, Sidebar, Dialog, Toast use document/window. All must be guarded with onMounted() or typeof window !== 'undefined'. Audit all current components.
  • Teleport — Dialog's <Teleport> requires #portal-target to exist in the DOM. In SSR, this may not be available during hydration. Use defer on Teleport or document the ClientOnly wrapping requirement.
  • QrCode, Video, LightboxImage — all require client-side initialisation. Must be explicitly documented.

prefers-reduced-motion

Every animated component must respect the user's motion preference using Tailwind's motion-reduce: variants:

/* applied to all transitions */
transition-all duration-300 motion-reduce:duration-0 motion-reduce:transition-none

Components affected: Dialog, Sidebar, Toast, Image (blur-up), FloatingPanel, Loader (spin/bounce/pulse), Placeholder (shimmer), all Vue <Transition> instances. This should be applied consistently across all components new and old.

Right-to-Left (RTL)

Not in scope now but decisions made today determine how hard it is later. Adopt Tailwind logical properties from the start on all new components:

  • Use ms-*/me-* instead of ml-*/mr-*
  • Use start-0/end-0 instead of left-0/right-0 for positioned elements
  • FloatingPanel start/end placements instead of left/right

Bundle size tracking

Add size-limit or bundlesize as a dev dependency. Set a budget: ESM bundle ≤ 100KB pre-gzip, ≤ 40KB post-gzip. Run in CI so size regressions are caught before merge.

Versioning strategy

Currently pre-1.0 (v0.0.x). Before 1.0, breaking changes are acceptable with minor bumps. After 1.0: strict semver, breaking changes require a major bump and migration guide.

ℹ
Recommendation: Target 1.0 after Phase 2 (Tier 1 components) is complete. Stabilise the icon API, Form context pattern, and slot naming conventions before 1.0 — these are the most disruptive things to change later.

9 UX Laws Applied

The specific application of each law to OUI — not abstract definitions, but concrete decisions.
Fitts's Law
All interactive elements ≥ 44×44px touch target. Button small variant must be audited. Pagination page buttons and Sidebar nav items must be adequately sized for thumb targets on mobile.
Hick's Law
Limit prop variants — use sensible defaults. Dropdown menus capped at ~7 visible items; separators and labels for longer lists. Storybook stories show the most common use first.
Jakob's Law
Use conventional names: Tooltip not FloatingPanel[role=tooltip]. Tabs keyboard behaviour must match ARIA APG exactly — that is what developers expect from prior framework experience.
Miller's Law
Pagination shows ≤ 7 page numbers at once. Sidebar navigation groups should not exceed 7 items without a visual separator. Accordion groups: max 7 items per section.
Law of Proximity
Card slots (#header, #content, #footer) make spatial relationships explicit. FormField composition (label above, helper below, error below helper) encodes proximity into the component structure.
Law of Common Region
Card, Dialog, Sidebar, Alert, Toast, and Popover all use a distinct visual region (background, border, shadow) to signal "this content belongs together." Shared region → shared context.
Aesthetic-Usability Effect
Stats cards, Code blocks, Empty states, and Timeline must be polished. Users perceive visually refined components as more reliable. Ugly skeletons undermine trust in the data they represent.
Zeigarnik Effect
Progress and Stepper keep users engaged by making completion feel achievable. Toast action buttons ("Undo") allow users to resolve interrupted operations. Completion states must feel satisfying.
Von Restorff Effect
Error and warning states must be visually distinct via colour and icon — never colour alone. Colour-blind users cannot distinguish red from green. Use aria-live for dynamically appearing alerts.
Tesler's Law
OUI absorbs ARIA complexity of Dialog, Tabs, Menu, and Combobox. FormField absorbs ID-linking boilerplate. registerIcons() absorbs icon management. Consumers write simple code; OUI does the hard parts.
Goal Gradient Effect
Stepper and Progress must show both what is done and how close the end is. The visual treatment of "complete" steps should feel rewarding (filled icon, stronger colour). Active step clearly differentiated from pending.
Serial Position Effect
In Sidebar navigation, place the most important items first. In Toast, the most recent notification should be visually prominent. In Pagination, boundary pages (first/last) are always shown regardless of the collapsed range.

10 Developer Experience

A component library is a developer-facing product. DX is as important as the components themselves.

TypeScript coverage

  • All component props must have TypeScript types — no any.
  • All emitted events typed with defineEmits<{ (e: 'name', payload: Type): void }>.
  • All exposed methods (Dialog's open(), close(), isOpen) typed with defineExpose.
  • Export all prop type interfaces: import type { ButtonProps } from 'ocelot-ui'.
  • Export IconProp = IconName | Component for use in consumer code.

IDE integration

  • JSDoc on props for VS Code / WebStorm IntelliSense descriptions.
  • Generate a components.d.ts at build time (via unplugin-vue-components) for auto-complete on <OuiButton> in consumer templates without explicit imports.

Development warnings

Components should console.warn in development (stripped in production) for:

  • Dialog: missing ariaLabel prop and no title slot content
  • Button iconOnly: missing aria-label
  • Image / LightboxImage: missing alt prop
  • Avatar: src provided but alt is empty on a non-decorative avatar
  • Icon: unrecognised string icon name (not in registry)

Documentation standards (per Storybook doc page)

  1. When to use / When not to use — which pattern to pick, not just how to use a given component
  2. Props table — auto-generated + manually edited descriptions
  3. Slots table — name, expected content, relationship to props
  4. Events table — event name, payload type, when it fires
  5. Keyboard interaction table — every key, what it does (required for interactive components)
  6. ARIA roles and attributes — what the component applies and why
  7. Related components — links to commonly used companions
  8. Accessibility notes — manual checks beyond automated Axe
